# Analytics Tracking – High-Level Spec
(GA4 + Mixpanel)

## 1. Goal

Establish a reliable, governed analytics system that captures **meaningful product behavior** across the platform, supports both anonymous and authenticated users, and enables clear decision-making.

Analytics must be:
- Deterministic
- Environment-safe
- Event-driven
- Ownership-enforced

This system explicitly avoids noisy, unstructured tracking.

---

## 2. Principles

- **Events over clicks** – Only track behavior that represents intent or value.
- **Canonical before tools** – Event definitions come before GA4 or Mixpanel implementation.
- **One source of truth** – A single event contract feeds all analytics tools.
- **Separation of concerns** – GA4 for traffic, Mixpanel for product behavior.
- **Governance first** – No analytics connection without environment validation.

---

## 3. Tool Responsibilities

### GA4 (Web & Traffic Analytics)
Used for:
- Page views
- Sessions
- Acquisition & referrers
- Returning vs new users
- High-level conversion visibility

### Mixpanel (Product Analytics)
Used for:
- User journeys
- Course / lesson engagement
- PDF usage
- Chat & AI interactions
- Registration funnels
- Retention & cohorts

Only a **minimal shared subset of events** is sent to both tools.

---

## 4. Canonical Event Model

All analytics events are defined canonically and emitted through a single tracking interface.

### Event Rules
- Every event has:
  - A fixed name
  - A strict property schema
  - A defined destination (GA4 / Mixpanel / Both)
- No direct SDK calls are allowed outside the analytics layer.
- No ad-hoc events are permitted.

---

## 5. Top 10 Canonical Product Events

| Event Name | Description | Destination |
|-----------|------------|-------------|
| `page_view` | Any page render | GA4 + Mixpanel |
| `session_started` | New user session | GA4 + Mixpanel |
| `user_identified` | Anonymous → identified transition | Mixpanel |
| `course_entered` | User enters a course | Mixpanel |
| `lesson_started` | Lesson consumption begins | Mixpanel |
| `lesson_completed` | Lesson completed | Mixpanel |
| `pdf_viewed` | PDF content consumed | Mixpanel |
| `chat_message_sent` | User sends chat message | Mixpanel |
| `registration_prompt_shown` | Soft gate displayed | Mixpanel |
| `registration_completed` | Successful registration | GA4 + Mixpanel |

Each event has a **strict property contract** defined in the Analytics Canonical Spec.

---

## 6. User Identity Handling

- All users start as anonymous.
- Anonymous users are tracked with a temporary identifier.
- Upon registration or login:
  - `user_identified` event is emitted
  - Anonymous history is aliased to the persistent user ID (Mixpanel)
- GA4 relies on session-based continuity only.

---

## 7. Environment Governance

Analytics must never be connected blindly.

Before enabling analytics, the task owner must confirm:
- Environment: `production` / `staging`
- Domain and final URL
- Correct GA4 property ID
- Correct Mixpanel project token

### Environment Rules
- Production analytics are disabled by default.
- Feature flags control analytics activation.
- Staging events must never pollute production dashboards.

---

## 8. Validation & Quality Rules

- No duplicate events per action.
- Page views must not double-fire on navigation.
- Events must appear only in the correct environment dashboards.
- Event payloads must match the canonical schema.
- Debug mode must allow inspection without emission.

---

## 9. Acceptance Criteria

- All 10 canonical events are visible in their designated tools.
- Page views and sessions behave correctly across navigation.
- Registration funnel is fully observable.
- Anonymous → identified user flow is verifiable.
- Analytics setup is confirmed as launch-safe.

---

## 10. Non-Goals

- No heatmaps or session replays in this phase.
- No auto-tracking of scrolls, hovers, or generic clicks.
- No agentic or ML-driven analytics workflows.
- No "track everything" behavior.

---

## 11. Ownership

Analytics is a shared responsibility between:
- Product
- Engineering
- Analytics owner (Ido)

No changes to analytics are allowed without updating the canonical spec.
